# Q45. Customers and Total Revenue

**Category:** RIGHT JOIN
**Difficulty:** Medium

---

## Problem Statement

The finance team wants a report showing every customer and the total revenue generated from their orders. Every customer should appear in the report, even if they have never placed an order.

## Objective

Return all customers along with their total revenue, ensuring customers with no orders appear with a revenue of 0.00.

## Tables Used

- `order_details`
- `orders`
- `customers`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| contact_name | Name of the customer contact |
| total_revenue | Total discounted revenue generated by this customer (0.00 if no orders placed) |

**Sample output:**

| contact_name | total_revenue |
|--------------|---------------|
| Maria Anders | 12543.65 |
| Thomas Hardy | 11624.90 |
| Ana Trujillo | 0.00 |

*(Sample values are illustrative, based on the standard Northwind dataset.)*

## Concepts Used

- RIGHT JOIN (chained, two levels)
- GROUP BY
- Aggregate Functions (SUM)
- NULL Handling (COALESCE)
- ROUND / CAST

## Why This Approach

**Why two `RIGHT JOIN`s instead of one:** the revenue data lives in `order_details`, but the chain from `order_details` to `customers` requires going through `orders`. Two joins are needed: `order_details ŌåÆ orders ŌåÆ customers`. Both must be `RIGHT JOIN` to preserve the right-most table (`customers`) all the way through the chain.

**Understanding the chain direction:** the join chain reads right-to-left in terms of preservation:
- First `RIGHT JOIN`: `order_details` (left) RIGHT JOIN `orders` (right) ŌĆö preserves all orders, even if they have no line items in `order_details` (edge case).
- Second `RIGHT JOIN`: `[order_details + orders]` (left) RIGHT JOIN `customers` (right) ŌĆö preserves all customers, even if they have no matching orders.

The net effect is that `customers` is fully preserved ŌĆö every customer appears regardless of order history.

**The LEFT JOIN equivalent (same result, arguably clearer to read):**
```sql
SELECT c.contact_name,
    COALESCE(ROUND(SUM(od.quantity * od.unit_price * (1 - od.discount))::numeric, 2), 0) AS total_revenue
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.customer_id
LEFT JOIN order_details od ON od.order_id = o.order_id
GROUP BY c.contact_name, c.customer_id;
```

This is structurally identical to Q31, which used `LEFT JOIN`. The RIGHT JOIN version reverses the table order in `FROM`/`JOIN` and flips join direction, but preserves exactly the same entity: `customers`.

**Why `COALESCE` is required:** `SUM()` returns NULL for customers with no orders ŌĆö not 0. `COALESCE` converts that NULL to the `0.00` the business requirement asks for.

**Why `::numeric` cast before `ROUND()`:** the revenue expression produces `double precision` due to the `discount` column. PostgreSQL's two-argument `ROUND()` only accepts `numeric`, so the explicit cast is required.

## Common Mistakes

- Making either `RIGHT JOIN` an `INNER JOIN`, which would drop the customers it was designed to preserve ŌĆö the same chain-integrity rule as chained `LEFT JOIN`s in Q31/Q39, just in the opposite direction.
- Forgetting `COALESCE` ŌĆö `SUM()` returns NULL (not 0) for empty groups.
- Grouping by `contact_name` alone instead of including `customer_id` ŌĆö risks silent merges for customers with identical contact names.
- Forgetting `::numeric` before `ROUND()`, causing a PostgreSQL type error.

**Medium**

## Interview Follow-up Questions

**1. How is this query the RIGHT JOIN equivalent of Q31 (Customers and Total Revenue using LEFT JOIN)?**

Q31 placed `customers` as the left (driving) table and chained two `LEFT JOIN`s through `orders` and `order_details`. This query places `customers` as the right (preserved) table and chains two `RIGHT JOIN`s from `order_details` through `orders`. Both preserve `customers` completely and produce identical results. The structural difference is purely the direction of traversal ŌĆö the entity being preserved, the join conditions, and the aggregation logic are all the same.

**2. In a chained RIGHT JOIN, which table is ultimately preserved?**

The rightmost table in the final `RIGHT JOIN` ŌĆö in this case `customers`. Think of it this way: each `RIGHT JOIN` says "preserve the right side." In a chain of two `RIGHT JOIN`s, the first preserves `orders` relative to `order_details`, but then the second `RIGHT JOIN` preserves `customers` relative to the combined `[order_details + orders]` result. The end effect is that `customers` drives the final result set ŌĆö every customer appears.

**3. Would this query give different results from Q31 if some customers had no orders?**

No ŌĆö both Q31 and Q45 produce identical results because they preserve the same entity (`customers`) using different syntactic approaches (`LEFT JOIN` vs `RIGHT JOIN` with reversed table order). The difference is stylistic and readability-based, not semantic. Some developers prefer the `LEFT JOIN` version because the driving table (`customers`) is visually first in the `FROM` clause, making the intent clearer at a glance.

**4. How would you add an order count column alongside total revenue?**

Add `COUNT(DISTINCT o.order_id)` to the `SELECT`:

```sql
SELECT
    c.contact_name,
    COUNT(DISTINCT o.order_id) AS order_count,
    COALESCE(ROUND(SUM(od.quantity * od.unit_price * (1 - od.discount))::numeric, 2), 0) AS total_revenue
FROM order_details od
RIGHT JOIN orders o ON o.order_id = od.order_id
RIGHT JOIN customers c ON c.customer_id = o.customer_id
GROUP BY c.contact_name, c.customer_id;
```

`COUNT(DISTINCT o.order_id)` is used rather than plain `COUNT(o.order_id)` because joining `order_details` multiplies order rows (one per line item) ŌĆö without `DISTINCT`, each order would be counted once per line item, inflating the order count.

**5. How would you find customers with revenue above the average using this query as a base?**

Wrap this query in a CTE and filter on the average:

```sql
WITH customer_revenue AS (
    SELECT
        c.contact_name,
        COALESCE(ROUND(SUM(od.quantity * od.unit_price * (1 - od.discount))::numeric, 2), 0) AS total_revenue
    FROM order_details od
    RIGHT JOIN orders o ON o.order_id = od.order_id
    RIGHT JOIN customers c ON c.customer_id = o.customer_id
    GROUP BY c.contact_name, c.customer_id
)
SELECT *
FROM customer_revenue
WHERE total_revenue > (SELECT AVG(total_revenue) FROM customer_revenue)
ORDER BY total_revenue DESC;
```

## Learning Outcomes

- Understand how chained `RIGHT JOIN`s work ŌĆö the rightmost table in the final join is the one ultimately preserved, regardless of what happens in earlier joins in the chain.
- Confirm that chained `RIGHT JOIN` and chained `LEFT JOIN` are syntactic mirrors of each other ŌĆö identical logic, different table ordering in the query.
- Reinforce that `SUM()` always requires `COALESCE` for zero-default handling in outer join contexts, regardless of join direction.
