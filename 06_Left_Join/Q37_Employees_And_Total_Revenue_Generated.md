# Q37. Employees and Total Revenue Generated

**Category:** LEFT JOIN
**Difficulty:** Medium

---

## Problem Statement

The management team wants a report showing every employee and the total revenue they have generated from customer orders. Employees who have not handled any orders should also appear with a revenue of 0.

## Objective

Return all employees along with the total revenue they generated, ensuring employees with no orders appear with a revenue of 0.00.

## Tables Used

- `employees`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| employee_name | Full name of the employee (first + last) |
| total_revenue | Total discounted revenue generated by this employee (0.00 if no orders handled) |

**Sample output:**

| employee_name | total_revenue |
|---------------|---------------|
| Margaret Peacock | 232890.85 |
| Janet Leverling | 202812.84 |
| Nancy Davolio | 128945.75 |
| Andrew Fuller | 0.00 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- LEFT JOIN (chained, two levels)
- GROUP BY
- Aggregate Functions (SUM)
- NULL Handling (COALESCE)
- ROUND / CAST
- String Concatenation (CONCAT)

## Why This Approach

**Why two `LEFT JOIN`s:** revenue lives in `order_details`, but the path from `employees` to `order_details` requires going through `orders` first — `orders` holds `employee_id`, and `order_details` holds line-item revenue. Both joins must be `LEFT JOIN`: if either were converted to `INNER JOIN`, employees with no orders would be dropped at that step, violating the requirement.

**Why `COALESCE(ROUND(SUM(...)::numeric, 2), 0)`:** for an employee with no orders, the full `LEFT JOIN` chain produces NULL for all `order_details` columns. `SUM()` over an all-NULL group returns `NULL` — not `0`. `COALESCE` converts that `NULL` into the `0.00` the business requirement explicitly asks for. Unlike `COUNT()`, `SUM()` never auto-defaults to zero for empty groups.

**Why `CONCAT(e.first_name, ' ', e.last_name)`:** the expected output shows a single `employee_name` column. `CONCAT()` builds this combined display name cleanly, handling potential NULL name parts more gracefully than the `||` operator (which returns NULL if either operand is NULL, while `CONCAT` treats NULL as an empty string).

**Why `GROUP BY e.employee_id` only:** `employee_id` is the unique key — grouping on it alone is sufficient to guarantee one row per employee. The name columns (`first_name`, `last_name`) are consumed inside `CONCAT()` in the `SELECT` list, so PostgreSQL does not require them in `GROUP BY`. This is a cleaner approach than grouping on both name columns alongside the ID.

**Why `::numeric` cast before `ROUND()`:** PostgreSQL's two-argument `ROUND()` requires a `numeric` type. The revenue expression produces `double precision` due to the `discount` column — the explicit cast is required to avoid a type error.

## Common Mistakes

- Converting either join to `INNER JOIN`, which silently drops employees with no order history.
- Forgetting `COALESCE`, leaving `NULL` in the revenue column for employees with no orders instead of `0.00` — unlike `COUNT`, `SUM` never auto-returns zero.
- Using `||` concatenation instead of `CONCAT()` — if `first_name` or `last_name` were ever `NULL`, `||` would return a NULL employee name rather than a partial name.
- Forgetting `::numeric` before `ROUND()`, which causes a PostgreSQL type error on `double precision` input.
- Grouping by `first_name, last_name` instead of `employee_id`, risking silent merges for employees who share a name.

## Interview Follow-up Questions

**1. Why does this query use `COALESCE` when Q34 (Employees and Customers Served) did not?**

Q34 used `COUNT(DISTINCT o.customer_id)`, which returns `0` for empty groups. This query uses `SUM()`, which returns `NULL` for empty groups. `COALESCE` is required for `SUM()` but redundant for `COUNT()`. This distinction — `COUNT` auto-returns `0`, all other aggregates return `NULL` — is one of the most important SQL semantics points to understand when combining `LEFT JOIN` with aggregation.

**2. Why is it sufficient to `GROUP BY e.employee_id` only, when `first_name` and `last_name` are also logically used?**

Because both name columns are consumed inside the `CONCAT()` function in the `SELECT` list, not referenced as standalone non-aggregated columns. PostgreSQL only requires non-aggregated, non-functionally-dependent columns to appear in `GROUP BY`. Since `CONCAT(first_name, last_name)` wraps both names inside an expression, neither needs to appear separately in the `GROUP BY` clause — `employee_id` alone (as the primary key) is sufficient to uniquely identify each group.

**3. What is the difference between `CONCAT()` and `||` for string concatenation in PostgreSQL?**

Both concatenate strings, but they handle NULL differently. If either operand is NULL, the `||` operator returns NULL for the entire expression — so `NULL || ' ' || 'Fuller'` produces NULL, not `' Fuller'`. `CONCAT()` treats NULL operands as empty strings — so `CONCAT(NULL, ' ', 'Fuller')` returns `' Fuller'`. For name columns, which could theoretically be NULL, `CONCAT()` is the safer and more defensive choice.

**4. How would you modify this query to rank employees by revenue and identify who is above and below the team average?**

Use a CTE to compute per-employee revenue, then compute the average in a second CTE and join:

```sql
WITH employee_revenue AS (
    SELECT
        CONCAT(e.first_name, ' ', e.last_name) AS employee_name,
        COALESCE(ROUND(SUM(od.unit_price * od.quantity * (1 - od.discount))::numeric, 2), 0) AS total_revenue
    FROM employees e
    LEFT JOIN orders o ON o.employee_id = e.employee_id
    LEFT JOIN order_details od ON od.order_id = o.order_id
    GROUP BY e.employee_id
),
avg_revenue AS (
    SELECT AVG(total_revenue) AS avg_rev FROM employee_revenue
)
SELECT
    er.employee_name,
    er.total_revenue,
    CASE WHEN er.total_revenue > ar.avg_rev THEN 'Above Average'
         WHEN er.total_revenue = 0 THEN 'No Orders'
         ELSE 'Below Average' END AS performance
FROM employee_revenue er, avg_revenue ar
ORDER BY er.total_revenue DESC;
```

**5. How would you add a column showing each employee's revenue as a percentage of total company revenue?**

Use `SUM() OVER ()` (a window function with no partition) to compute the company-wide total alongside each row:

```sql
SELECT
    employee_name,
    total_revenue,
    ROUND((total_revenue / SUM(total_revenue) OVER () * 100)::numeric, 2) AS revenue_pct
FROM (
    SELECT
        CONCAT(e.first_name, ' ', e.last_name) AS employee_name,
        COALESCE(ROUND(SUM(od.unit_price * od.quantity * (1 - od.discount))::numeric, 2), 0) AS total_revenue
    FROM employees e
    LEFT JOIN orders o ON o.employee_id = e.employee_id
    LEFT JOIN order_details od ON od.order_id = o.order_id
    GROUP BY e.employee_id
) sub
ORDER BY total_revenue DESC;
```

## Learning Outcomes

- Understand why `SUM()` requires `COALESCE` for empty-group NULL handling while `COUNT()` does not — the most important aggregate behaviour distinction for `LEFT JOIN` queries.
- Learn why `GROUP BY primary_key` is sufficient when display columns are consumed inside expressions (`CONCAT`) rather than referenced as standalone `SELECT` items.
- Know the difference between `CONCAT()` and `||` NULL handling — a practical PostgreSQL-specific detail that prevents silent NULL name values.
