# Q31. Customers and Total Revenue

**Category:** LEFT JOIN
**Difficulty:** Medium

---

## Problem Statement

The sales team wants a report showing every customer and the total revenue they have generated. Customers with no orders should also appear in the report with revenue as 0.

## Objective

Return all customers along with their total revenue, ensuring customers with no orders still appear with a revenue value of 0.00 rather than being excluded or showing NULL.

## Tables Used

- `customers`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| contact_name | Name of the customer contact |
| revenue | Total discounted revenue generated by this customer (0.00 if no orders placed) |

**Sample output:**

| contact_name | revenue |
|--------------|---------|
| Maria Anders | 15234.50 |
| Ana Trujillo | 0.00 |
| Thomas Hardy | 11624.90 |
| Christina Berglund | 0.00 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format â€” not guaranteed to match your exact data instance.)*

## Concepts Used

- LEFT JOIN (chained, two levels)
- GROUP BY
- Aggregate Functions (SUM)
- NULL Handling (COALESCE)
- ROUND / CAST

## Why This Approach

**Why two `LEFT JOIN`s instead of one:** the revenue data lives in `order_details`, but there is no direct link between `customers` and `order_details` â€” `orders` is the bridge table. Two `LEFT JOIN`s are needed to chain `customers â†’ orders â†’ order_details`. Both must be `LEFT JOIN` (not `INNER JOIN`), because:
- If `LEFT JOIN customers â†’ orders` were an `INNER JOIN`, customers with no orders would be dropped at that step.
- If `LEFT JOIN orders â†’ order_details` were an `INNER JOIN`, any order with no line items (edge case in Northwind, but possible in production) would also be dropped, and more critically, the NULL row produced by the first `LEFT JOIN` would be eliminated â€” taking the no-order customers with it.

**Why `COALESCE` is required here:** for a customer with no orders, the entire chain through `orders` and `order_details` produces `NULL` values. `SUM()` over an all-`NULL` group returns `NULL`, not `0`. `COALESCE(SUM(...), 0)` converts that `NULL` into the literal `0.00` the business requirement explicitly asks for. This is the same principle as Q27 â€” `SUM` never auto-defaults to zero; `COALESCE` is always needed to make that explicit.

**Why `ROUND(...::numeric, 2)`:** PostgreSQL's `ROUND()` requires a `numeric` type argument when specifying decimal places. The revenue expression produces a `double precision` result due to the floating-point `discount` column, so the `::numeric` cast is required before `ROUND()` â€” without it, PostgreSQL raises a type error.

**Why `GROUP BY c.contact_name, c.customer_id`:** both columns are included because `customer_id` is the unique key that guarantees correct grouping, while `contact_name` is selected as a display column and therefore must also appear in `GROUP BY`. Grouping by `contact_name` alone risks merging two different customers who happen to share the same name.

## Common Mistakes

- Making either join an `INNER JOIN` â€” the first one silently drops no-order customers; the second one, even if the first is `LEFT`, can eliminate the NULL-padded rows produced by the first join, pulling no-order customers out of the result through the back door.
- Using `SUM(..., 0)` or expecting `SUM()` to return zero automatically for empty groups â€” it doesn't; `COALESCE` is always required for this.
- Grouping by `contact_name` only, which risks silently merging two customers with the same name into one row with combined revenue.
- Forgetting `::numeric` before `ROUND()` in PostgreSQL, which causes a type mismatch error on `double precision` inputs.

## Interview Follow-up Questions

**1. Why are both joins `LEFT JOIN` rather than making just the first one `LEFT JOIN`?**

Making only the first join `LEFT JOIN` and the second `INNER JOIN` would eliminate the no-order customers again at the second step. Here's why: the first `LEFT JOIN` correctly produces a NULL-padded row for customers with no orders. But an `INNER JOIN` on `order_details` then tries to match those NULL-padded rows â€” since `o.order_id` is `NULL` for those customers, there's nothing to match in `order_details`, and the `INNER JOIN` drops them entirely. Both joins must be `LEFT JOIN` to keep the chain intact all the way to the aggregation step.

**2. What is `COALESCE` doing here and why can't `SUM()` just default to 0?**

`COALESCE(value, fallback)` returns the first non-`NULL` argument. Here, for a customer with no orders, `SUM()` receives a group of all `NULL` values and returns `NULL` â€” not `0`. SQL aggregate functions, by definition, ignore `NULL` values when computing their result, and when every value in a group is `NULL`, the result itself is `NULL`. `COALESCE(..., 0)` is what explicitly converts that `NULL` output into the `0` the business requirement asks for. There is no way to configure `SUM()` to auto-default to zero; `COALESCE` is always the correct tool.

**3. Why does `GROUP BY` include `customer_id` if only `contact_name` is selected?**

`contact_name` is not guaranteed to be unique across customers â€” two different customers could share the same contact name. Grouping by `contact_name` alone would silently merge those customers into one row and combine their revenues, producing incorrect results. Including `customer_id` in `GROUP BY` guarantees each row represents exactly one customer. PostgreSQL also requires every non-aggregated column in the `SELECT` list to appear in `GROUP BY`, so even if names were unique, both columns would need to be included.

**4. Why is `::numeric` required inside `ROUND()` in PostgreSQL?**

PostgreSQL's `ROUND(value, decimal_places)` function is overloaded â€” the two-argument version only accepts `numeric`, not `double precision`. The revenue expression `unit_price * quantity * (1 - discount)` involves the `discount` column, which is stored as `double precision` (a floating-point type). Without the explicit `::numeric` cast, PostgreSQL raises: `ERROR: function round(double precision, integer) does not exist`. The cast converts the result to a fixed-precision decimal type that `ROUND()` accepts cleanly.

**5. How would you extend this to also show each customer's order count alongside revenue?**

Add `COUNT(DISTINCT o.order_id)` to the `SELECT` list. Using `DISTINCT` is essential here â€” without it, every `order_details` row for a given order would be counted as a separate order, massively inflating the count:

```sql
SELECT
    c.contact_name,
    COUNT(DISTINCT o.order_id) AS order_count,
    COALESCE(ROUND(SUM(od.unit_price * od.quantity * (1 - od.discount))::numeric, 2), 0) AS revenue
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
LEFT JOIN order_details od ON o.order_id = od.order_id
GROUP BY c.contact_name, c.customer_id;
```

## Learning Outcomes

- Understand why chained `LEFT JOIN`s must remain `LEFT JOIN` throughout the chain â€” converting any intermediate join to `INNER JOIN` silently drops the very rows the outer join was designed to preserve.
- Internalize that `SUM()` returns `NULL` (not `0`) for empty groups, and that `COALESCE` is always the explicit, correct fix when a zero default is required.
- Recognise the PostgreSQL-specific `::numeric` cast requirement inside `ROUND()` â€” a common type error when first writing revenue calculations in PostgreSQL.
- Practice grouping on the primary key alongside display columns to avoid silent row merges from non-unique display values.
