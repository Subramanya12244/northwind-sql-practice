# Q39. Categories and Total Revenue

**Category:** LEFT JOIN
**Difficulty:** Medium

---

## Problem Statement

The management team wants a report showing every product category and the total revenue generated by products in that category. Categories with no sales should also appear with a revenue of 0.

## Objective

Return all categories along with the total revenue generated by their products, ensuring categories with no sales appear with a revenue of 0.00.

## Tables Used

- `categories`
- `products`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| category_name | Name of the product category |
| total_revenue | Total discounted revenue generated by products in this category (0.00 if no sales) |

**Sample output:**

| category_name | total_revenue |
|---------------|---------------|
| Beverages | 267868.15 |
| Dairy Products | 234507.29 |
| Meat/Poultry | 163022.36 |
| Seafood | 0.00 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- LEFT JOIN (chained, two levels)
- GROUP BY
- Aggregate Functions (SUM)
- NULL Handling (COALESCE)
- ROUND / CAST

## Why This Approach

**Why two `LEFT JOIN`s:** revenue lives in `order_details`, but there is no direct link from `categories` to `order_details`. `products` is the required bridge — it maps `category_id` to `product_id`, which `order_details` then references. Both joins must be `LEFT JOIN`: the first preserves categories with no products; the second preserves categories whose products have never been ordered. Converting either to `INNER JOIN` silently drops those categories from the result.

**Why `COALESCE(ROUND(SUM(...)::numeric, 2), 0)`:** for a category with no sales, `SUM()` over an all-NULL group returns `NULL` — not `0`. `COALESCE` converts that `NULL` into the `0.00` the business requirement explicitly asks for. This is the same pattern as Q31 and Q37 — `SUM` always requires `COALESCE` for zero-default handling, unlike `COUNT` which auto-returns `0`.

**Why `GROUP BY c.category_name, c.category_id`:** `category_id` is the unique key; `category_name` must also appear as a non-aggregated selected column. Grouping on name alone risks merging categories with identical names.

## Common Mistakes

- Converting either join to `INNER JOIN` — the first drops categories with no products; the second drops categories whose products exist but have no order history.
- Forgetting `COALESCE` — `SUM()` returns `NULL` for categories with no sales, not `0`.
- Skipping the `products` bridge table and attempting to join `categories` directly to `order_details` — there is no shared key, so this is impossible without the intermediate table.

**Medium**

## Interview Follow-up Questions

**1. Why are two LEFT JOINs required instead of one?**

The category-to-revenue path requires two hops: `categories → products` (via `category_id`) and `products → order_details` (via `product_id`). There is no direct foreign key between `categories` and `order_details`, so the `products` bridge table is mandatory. Each join must remain `LEFT JOIN` to preserve categories at both levels — those with no products at all, and those with products that have never been ordered.

**2. What would happen if the second join (products → order_details) were INNER JOIN?**

Categories that have products but none of those products have ever been ordered would lose their NULL-padded rows at that step. The INNER JOIN would eliminate them from the result, even though the first LEFT JOIN correctly preserved them. The end result would look like "categories with at least one ordered product" — a different, narrower query from what the business asked for.

**3. Why does this query need `COALESCE` when Q35 (Categories and Number of Products) did not?**

Q35 used `COUNT()`, which returns `0` for empty groups. This query uses `SUM()`, which returns `NULL` for empty groups. The difference is fundamental to SQL aggregate semantics: `COUNT` is the only standard aggregate that auto-returns `0`; all others (`SUM`, `AVG`, `MIN`, `MAX`) return `NULL` when there are no non-NULL values to aggregate. `COALESCE` is only ever required for the latter group.

**4. How would you extend this to also show each category's average product price alongside total revenue?**

Add `ROUND(AVG(p.unit_price)::numeric, 2)` to the `SELECT`. For categories with no products, `AVG()` returns `NULL` — wrap in `COALESCE(..., 0)` if a zero default is needed:

```sql
SELECT
    c.category_name,
    COALESCE(ROUND(SUM(od.unit_price * od.quantity * (1 - od.discount))::numeric, 2), 0) AS total_revenue,
    COALESCE(ROUND(AVG(p.unit_price)::numeric, 2), 0) AS avg_product_price
FROM categories c
LEFT JOIN products p ON p.category_id = c.category_id
LEFT JOIN order_details od ON od.product_id = p.product_id
GROUP BY c.category_name, c.category_id
ORDER BY total_revenue DESC;
```

**5. How would you show each category's revenue as a percentage of total company revenue?**

Use `SUM() OVER ()` as a window function to compute the company-wide total alongside each category row, then divide:

```sql
WITH category_revenue AS (
    SELECT
        c.category_name,
        COALESCE(SUM(od.unit_price * od.quantity * (1 - od.discount)), 0) AS total_revenue
    FROM categories c
    LEFT JOIN products p ON p.category_id = c.category_id
    LEFT JOIN order_details od ON od.product_id = p.product_id
    GROUP BY c.category_name, c.category_id
)
SELECT
    category_name,
    ROUND(total_revenue::numeric, 2) AS total_revenue,
    ROUND((total_revenue / NULLIF(SUM(total_revenue) OVER (), 0) * 100)::numeric, 2) AS revenue_pct
FROM category_revenue
ORDER BY total_revenue DESC;
```

`NULLIF(..., 0)` in the denominator guards against a division-by-zero error if total company revenue were somehow zero.

## Learning Outcomes

- Understand why both joins in a chained `LEFT JOIN` path must remain `LEFT JOIN` — converting any intermediate join to `INNER JOIN` silently undoes the outer join's preservation work.
- Reinforce that `SUM()` requires `COALESCE` for empty-group handling while `COUNT()` does not — understanding this distinction eliminates an entire class of NULL-related report errors.
- Practice multi-table `LEFT JOIN` chains through a mandatory bridge table, a pattern that recurs across virtually every revenue-by-dimension report in real-world analytics.
