# Q20. Top 5 Customers by Revenue Using CTE

**Category:** Common Table Expressions (CTEs)
**Difficulty:** Easy

---

## Problem Statement

Restating Q3's business need using an explicit CTE rather than a direct aggregation query, to demonstrate the CTE pattern for simple top-N reporting.

## Objective

Use a CTE to calculate per-customer revenue, then return the top 5 customers by revenue from that CTE.

## Tables Used

- `customers`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| customer_id | Unique identifier of the customer |
| company_name | Name of the customer's company |
| total_revenue | Sum of discounted revenue generated by this customer |

**Sample output:**

| customer_id | company_name | total_revenue |
|-------------|--------------|---------------|
| QUICK | QUICK-Stop | 117483.39 |
| ERNSH | Ernst Handel | 104874.98 |
| SAVEA | Save-a-lot Markets | 104361.95 |
| RATTC | Rattlesnake Canyon Grocery | 51097.80 |
| HUNGO | Hungry Owl All-Night Grocers | 49979.91 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- CTE
- INNER JOIN
- GROUP BY
- Aggregate Functions (SUM)
- ORDER BY
- LIMIT

## Why This Approach

**Why use a CTE here at all, given Q3 solved the identical problem without one:** functionally there is no difference for this simple case — the CTE version is purely a style/readability choice. It becomes genuinely valuable once a query needs to *reuse* the same aggregated result multiple times (e.g. for both display and a comparison, as in Q6/Q19) — this question establishes the pattern in its simplest form before later questions build on it.

## Common Mistakes

- Believing the CTE version is inherently 'better' or faster than the inline version — for a single-use case like this, PostgreSQL's planner typically treats them equivalently; the value here is purely readability and future extensibility.
- Forgetting `ORDER BY` before `LIMIT`, same risk as Q3.

## Difficulty

**Easy**

## Interview Follow-up Questions

1. For this specific query, is there any performance difference between using a CTE and writing the aggregation inline? Why might a CTE still be preferable stylistically?
2. At what point does a CTE stop being 'just style' and start mattering functionally?
3. How would you extend this CTE to also calculate each customer's rank, not just filter to the top 5?

## Learning Outcomes

- Understand that CTEs are primarily a readability/composability tool, not always a performance optimization, for simple single-use cases.
- Establish the foundational CTE pattern that more advanced questions (Q21-Q24) build upon.
