# Q19. Customers Above Average Revenue Using CTE

**Category:** Common Table Expressions (CTEs)
**Difficulty:** Medium

---

## Problem Statement

Restating Q6's business need with an explicit two-CTE structure — one CTE for customer revenue, a second CTE purely for the overall average — to demonstrate clean CTE composition as a readability pattern.

## Objective

Use a CTE to calculate per-customer revenue, and a second CTE to calculate the overall average, then return customers exceeding that average.

## Tables Used

- `customers`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| customer_id | Unique identifier of the customer |
| company_name | Name of the customer's company |
| total_revenue | Sum of discounted revenue generated by this customer |

**Sample output:**

| customer_id | company_name | total_revenue |
|-------------|--------------|---------------|
| QUICK | QUICK-Stop | 117483.39 |
| ERNSH | Ernst Handel | 104874.98 |
| SAVEA | Save-a-lot Markets | 104361.95 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- CTE (multiple)
- INNER JOIN
- GROUP BY
- Aggregate Functions (SUM, AVG)
- Cross Join (implicit, scalar)

## Why This Approach

**Why a second, dedicated CTE (`overall_avg`) instead of a scalar subquery in `WHERE` (as in Q6):** both approaches are functionally equivalent here, but separating the average into its own named CTE makes the query's intent more self-documenting — `overall_avg` reads clearly as 'the single company-wide benchmark value' independent of the customer-level detail in `customer_revenue`.

**Why `FROM customer_revenue cr, overall_avg oa`:** since `overall_avg` produces exactly one row (a single scalar average), pairing it with `customer_revenue` via a comma-join (an implicit cross join) attaches that one constant value to every row of `customer_revenue` without needing an explicit `ON` condition — there's nothing to join *on*, since it's a single scalar being broadcast to every row.

## Common Mistakes

- Adding an explicit `JOIN ... ON` condition to `overall_avg` when none is needed or possible (it has no shared key with `customer_revenue` — it's a single scalar row).
- Forgetting that if `overall_avg` ever returned more than one row, this comma-join would silently multiply rows in the result — fine here, but a sign to double check `overall_avg`'s `SELECT` doesn't accidentally include a `GROUP BY`.

## Difficulty

**Medium**

## Interview Follow-up Questions

1. Why does pairing `customer_revenue` and `overall_avg` with a comma not produce duplicate rows here?
2. What would happen if `overall_avg`'s CTE definition accidentally included a `GROUP BY` clause?
3. How does this two-CTE structure compare in clarity to Q6's single-CTE-plus-subquery version? Which would you choose for a teammate to maintain?

## Learning Outcomes

- Practice composing multiple CTEs where one computes a detail-level result and another computes a single benchmark value.
- Understand implicit cross joins (`FROM a, b`) and when they're safe to use — specifically, when one side is guaranteed to return exactly one row.
