# Q22. Top Employee by Revenue Using 3 CTEs

**Category:** Common Table Expressions (CTEs)
**Difficulty:** Medium

---

## Problem Statement

Leadership wants to identify the single highest-revenue employee company-wide, with the requirement that ties be handled correctly — if two employees are exactly tied for first, both should be returned, not an arbitrary one.

## Objective

Find the employee(s) with the single highest total revenue, using a layered three-CTE structure: aggregate, find the maximum, then filter to matches.

## Tables Used

- `employees`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| employee_id | Unique identifier of the top-revenue employee |
| first_name | Employee's first name |
| last_name | Employee's last name |
| total_revenue | Total discounted revenue generated by this employee (the company-wide maximum) |

**Sample output:**

| employee_id | first_name | last_name | total_revenue |
|-------------|------------|-----------|---------------|
| 4 | Margaret | Peacock | 232890.85 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- CTE (multiple/chained)
- GROUP BY
- Aggregate Functions (SUM, MAX)
- Tie Handling
- INNER JOIN

## Why This Approach

**Why three CTEs instead of a simple `ORDER BY ... LIMIT 1`:** `LIMIT 1` would silently return just one row even if two employees are exactly tied for the highest revenue — failing the explicit 'handle ties correctly' requirement. Comparing directly against `MAX(total_revenue)` instead naturally returns *every* employee whose revenue equals that maximum, ties included, with no extra logic required.

**Why chain three CTEs rather than two:** `employee_revenue` aggregates revenue per employee; `max_revenue` reduces that to a single scalar (the company-wide highest); `ranked_employees` joins the two together on equality, naturally catching every tie. Each CTE has a single, clear responsibility, which makes the query easy to read top-to-bottom and easy to debug by inspecting any one CTE in isolation.

## Common Mistakes

- Using `ORDER BY total_revenue DESC LIMIT 1`, which breaks the explicit tie-handling requirement by returning only one row even when multiple employees are tied.
- Using `RANK() = 1` instead of comparing to `MAX()` directly — both work correctly here, but `MAX()`-based equality is arguably simpler when you only care about the single top group, not a full ranking.

**Medium**

## Interview Follow-up Questions

1. Why is `LIMIT 1` the wrong tool here, given the tie-handling requirement?
2. Walk through what each of the three CTEs is responsible for, and why splitting them this way aids readability.
3. How would you adapt this to find the top employee per *region* rather than company-wide?
4. Could this be solved with `RANK()` instead of comparing to `MAX()`? What would change?

## Learning Outcomes

- Understand why naive `LIMIT`-based 'top 1' queries silently fail tie-handling requirements, and how to fix that with an equality-based `MAX()` comparison.
- Practice chaining multiple CTEs, each with a single clear responsibility, to build a readable multi-step query.
