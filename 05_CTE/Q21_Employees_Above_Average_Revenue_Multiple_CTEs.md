# Q21. Employees Above Average Revenue Using Multiple CTEs

**Category:** Common Table Expressions (CTEs)
**Difficulty:** Medium

---

## Problem Statement

Restating Q7's business need using a two-CTE structure (employee revenue, then average revenue), joined together with an explicit `JOIN ... ON` condition that doubles as the filter.

## Objective

Calculate employee revenue in one CTE, average revenue in a second CTE, and join them using the comparison itself as the join condition.

## Tables Used

- `employees`
- `orders`
- `order_details`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| employee_id | Unique identifier of the employee |
| first_name | Employee's first name |
| last_name | Employee's last name |
| total_revenue | Sum of discounted revenue generated by this employee |

**Sample output:**

| employee_id | first_name | last_name | total_revenue |
|-------------|------------|-----------|---------------|
| 4 | Margaret | Peacock | 232890.85 |
| 3 | Janet | Leverling | 202812.84 |
| 1 | Nancy | Davolio | 192107.60 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- CTE (multiple)
- INNER JOIN (non-equality condition)
- GROUP BY
- Aggregate Functions (SUM, AVG)

## Why This Approach

**Why `JOIN avg_revenue ar ON er.total_revenue > ar.avg_rev` is valid and useful:** SQL `JOIN ... ON` conditions aren't restricted to equality — any boolean expression is valid. Since `avg_revenue` produces exactly one row, this join condition effectively performs the filter directly within the join step rather than via a separate `WHERE` clause — functionally identical to Q19's `WHERE` version, but demonstrating that joins can encode inequality comparisons, not just equality lookups.

**Why this only works safely because `avg_revenue` is single-row:** if it returned multiple rows, this `JOIN ... ON >` condition would behave like a non-equi join, potentially producing a cross-product-like explosion of rows where each employee row is matched against every qualifying average row.

## Common Mistakes

- Using this same `JOIN ... ON >` pattern against a multi-row CTE, not realizing it could create unwanted row multiplication.
- Assuming `JOIN` conditions must always be equality — a common but incorrect assumption for SQL beginners.

## Difficulty

**Medium**

## Interview Follow-up Questions

1. Is `JOIN ... ON >` valid SQL? What would happen if the right-hand CTE returned more than one row?
2. How is this functionally different from filtering with `WHERE` after a `JOIN ... ON true` or cross join?
3. Would you recommend this pattern in production code, or is `WHERE` clearer? Justify your answer.

## Learning Outcomes

- Learn that `JOIN ... ON` supports arbitrary boolean conditions, not just equality — and understand the risk this introduces if cardinality assumptions are wrong.
- Continue building fluency with multi-CTE composition for 'compare to benchmark' queries.
