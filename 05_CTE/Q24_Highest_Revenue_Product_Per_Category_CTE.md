# Q24. Highest Revenue Product per Category (CTE + Window Function)

**Category:** Common Table Expressions (CTEs)
**Difficulty:** Hard

---

## Problem Statement

Identical business requirement to Q14 — for every category, find the top-revenue product(s), correctly handling ties — placed here in the CTE section to highlight the same pattern as a CTE/window-function technique in its own right.

## Objective

For every category, return the product(s) generating the highest revenue within that category, including all ties.

## Tables Used

- `products`
- `order_details`
- `categories`

## Expected Output

| Column Name | Description |
|-------------|-------------|
| category_id | Unique identifier of the category |
| category_name | Name of the category |
| product_id | Unique identifier of the top-revenue product in this category |
| product_name | Name of the top-revenue product |
| total_revenue | Total discounted revenue generated by this product |

**Sample output:**

| category_id | category_name | product_id | product_name | total_revenue |
|-------------|---------------|------------|--------------|---------------|
| 1 | Beverages | 38 | Côte de Blaye | 149984.20 |
| 2 | Condiments | 63 | Vegie-spread | 21477.80 |
| 4 | Dairy Products | 59 | Raclette Courdavault | 76683.75 |
| 6 | Meat/Poultry | 29 | Thüringer Rostbratwurst | 87736.40 |

*(Sample values are illustrative, based on the standard Northwind dataset, and intended to show shape/format — not guaranteed to match your exact data instance.)*

## Concepts Used

- CTE (multiple)
- Window Functions (RANK)
- PARTITION BY
- GROUP BY
- Aggregate Functions (SUM)
- Tie Handling
- INNER JOIN

## Why This Approach

**Identical reasoning to Q14, with one addition — the join to `categories`:** Q14 returned `category_id` as a raw foreign key; this version joins back to `categories` to also display `category_name`, making the output friendlier for a non-technical stakeholder reading a report rather than a raw query result.

**Why this still belongs in the 'CTE' section conceptually:** while Q14 lives under 'Category & Product Analytics' as a business question, the technique itself — chaining an aggregation CTE into a ranking CTE — is exactly the kind of reusable CTE pattern this section is meant to teach, regardless of which business entity it's applied to.

## Common Mistakes

- Same risks as Q14: using `ROW_NUMBER()` instead of `RANK()`, or omitting `PARTITION BY category_id`.
- Joining `categories` before aggregation, which doesn't change the result here (categories has no multiplying effect) but is worth being deliberate about regardless.

## Difficulty

**Hard**

## Interview Follow-up Questions

1. How is this query different from Q14, if at all? What does the extra `categories` join add?
2. Why does adding `category_name` not change the underlying ranking logic?
3. If `categories` had duplicate `category_id` values (a data quality problem), how would that affect this query?

## Learning Outcomes

- Reinforce the 'top N per group with ties' window function pattern from Q14 by applying it again with an extra presentation-layer join.
- Understand that adding descriptive columns from a dimension table after the ranking step doesn't change the analytical result, only its readability.
